# Flujo GTD

## 1. Capturar

- Todo entra primero en `00 Inbox/`.
- Cada entrada debe ser una nota corta con un titulo claro.
- Si ya sabes que es una tarea ejecutable, puedes crearla directo en `01 GTD/Tasks/`.

## 2. Aclarar

Al procesar una entrada decide una sola cosa:

- `task`: accion concreta.
- `project`: requiere mas de una accion.
- `resource`: material de consulta.
- `note`: idea elaborada o nota permanente.
- `media`: item de consumo cultural.
- `archive`: no accionable ni util ahora.

## 3. Organizar

### Tareas

Estados recomendados:

- `inbox`
- `next`
- `waiting`
- `scheduled`
- `someday`
- `done`
- `cancelled`

### Proyectos

Estados recomendados:

- `active`
- `on-hold`
- `review`
- `done`
- `archived`

### Media

Estados recomendados:

- `backlog`
- `in-progress`
- `paused`
- `completed`
- `dropped`

## 4. Revisar

Revision semanal minima:

1. Vaciar `00 Inbox/`.
2. Confirmar que cada proyecto activo tenga al menos una tarea `next`.
3. Limpiar tareas `waiting` y `scheduled` vencidas.
4. Mover cerrado o inactivo a `90 Archive/`.

## 5. Ejecutar

Trabaja desde la vista [[01 GTD/GTD.base|GTD.base]] o directamente desde la carpeta `01 GTD/Tasks/`.

Prioriza por:

1. contexto
2. energia
3. tiempo disponible
4. fecha real
5. prioridad

## Debilidades que se corrigen frente a Notion

- Se elimina dependencia de formulas y botones para operar.
- Los estados dejan de estar escondidos en vistas filtradas.
- Las relaciones se vuelven enlaces legibles y editables.
- La captura no depende de una base con columnas exactas.
- El sistema puede seguir funcionando offline y con git.
